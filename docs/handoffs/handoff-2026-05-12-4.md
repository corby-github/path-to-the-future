## Key Decisions and Why

### Arcade modal as a two-state overlay, not a fake temporary room
- **Decision:** `ArcadeModal` mounts inline in DecisionRoom, `phase: 'menu' | 'playing'` toggles. Playing state renders `MinigameByVariant`; minigame `onComplete` returns to menu so plays chain. Esc bails entirely.
- **Why:** Routing the minigame as a transient `MinigameRoom` config would require room-transition choreography for something that isn't a "month room." Modal-with-states keeps DecisionRoom authoritative.
- **Alternative:** Temporary `roomType` route. Rejected as too much plumbing.

### Universal interactables layer (`public/universal/interactables.json`)
- **Decision:** New parallel content path next to `public/careers/{packId}/`. Loader fetches both optionally and merges into `pack.interactables`. Pack-specific id wins on collision.
- **Why:** Arcade is universal by concept; duplicating it in every future pack is the wrong primitive. User explicitly redirected: *"no it should be universal — so lets put it where it belongs."*
- **Alternative:** Per-pack copies. Rejected — duplication.

### Closed `MinigameVariant` union for v1
- **Decision:** Keep `'blackjack' | 'code-review' | 'reaction-sprint' | 'pong' | 'forty-two'` as a closed string-literal union. Don't introduce a registry pattern.
- **Why:** Closed union forces every variant to surface in 4 specific call sites (`MinigameByVariant` switch, `MinigameRoom` switch, `progressSlice.lastArcadeXpAt` initial record, `MinigameReplayCard.VARIANT_LABELS`). Compiler tells you what's missing. Registry pattern is v2 multi-pack work — premature here.
- **Alternative:** Registry with per-pack variant maps. Rejected as v2-scope.

### Arcade plays don't record to `history.minigames`
- **Decision:** `recordMinigame` only dispatches when `mode === 'scheduled'`.
- **Why:** Replay timeline (§11.1) is the player's *what-happened-in-the-scheduled-moments* record. Arcade plays would dilute the narrative weight.
- **Alternative:** Record everything, filter at replay-render. Rejected — same hygiene problem deferred.

### Real-time hourly throttle, all rewards gated
- **Decision:** `ARCADE_THROTTLE_MS = 60 * 60 * 1000`. Per-variant. When eligible: XP + stat effects + savings all fire. When throttled: minigame runs for fun but dispatches nothing.
- **Why:** Real-time can't be grinded by month-clicking. Gating all rewards (not just XP) prevents savings/skill grinding via Blackjack / Code Review.
- **Alternative:** In-game time (rejected — grindable). XP-only gating (rejected — opens stat-grind door).

### "The Ultimate Question" rename hides the punchline
- **Decision:** Player-facing menu row "42" → "The Ultimate Question". Internal variant id stays `'forty-two'`. Answer option inside game stays `'42'`.
- **Why:** Showing "42" in the menu pre-spoils the joke. Title should telegraph the *question*, not the *answer*.
- **Alternative:** Rename the variant id too. Rejected — schema churn for cosmetic gain (would invalidate every save).

### Pong AI: capped speed + deadzone, no reaction-lag buffer
- **Decision:** `AI_MAX_SPEED = 360 v.u./sec`, `AI_DEADZONE = 6 px`. AI sees the ball's live position, not a buffered past position.
- **Why:** The 1000×600 court is small enough that ball crosses in <2s at top speed. Real reaction-lag implementation read sluggish — AI would consistently miss because the ball had already passed the buffered position. Speed cap + deadzone deliver the "beatable on a good day, frustrating on a bad one" feel without the plumbing.
- **Alternative:** Position-buffer with reaction-lag. Rejected on feel.

### Esc as the universal "give up" key inside minigames
- **Decision:** Pong + FortyTwo both treat Esc as forfeit during playing phase (routes through `handleContinue` with current score → classified as fail under the outcome rule).
- **Why:** Pong shipped without an exit key and the user immediately reported wanting one. Esc is the canonical bail-out. For Pong, classifies as fail (no win-score reached). For FortyTwo, also fail (no answer = wrong answer).
- **Alternative:** Skip the exit entirely. Rejected — bad UX for arcade plays the player wants to back out of.

## Dead Ends

- **DevPanel "spawn arcade" as a dedicated checkbox.** Assistant shipped it as a standalone checkbox in PR #50; user refactored to a one-shot option in the existing `trigger` dropdown during merge cleanup to keep the dev row from bloating. Cleanup also trimmed `forceArcade: boolean` from the `DevControls` interface but left consumers in `DecisionRoom` + `DevControlsProvider` — broke typecheck on main. Caught when starting PR #51 work, landed a separate `fix(dev)` commit restoring the interface field. **Lesson:** user's "trim as I merge" pattern means small follow-up fixes are part of the loop, not anomalies — and merging a PR with sweeping cleanup deserves a `npm run verify` on main before starting the next branch.
- **rAF chain dying after Pong goals.** First Pong PR shipped with `return` statements in the goal branches that skipped `raf = requestAnimationFrame(tick)` scheduling. Ball reset to center, never moved again. Caught by user's 5-minute playtest, not by typecheck or lint. **Lesson:** control-flow bugs in animation loops are invisible to static checks; the user's playtest is doing real QA work.
- **Invisible "Esc to walk away" hint.** ArcadeModal hint used `palette.surface` (mid-tone for cards/obstacles) against the modal's dimmed `rgba(20,20,20,0.55)` backdrop. Read as nearly invisible. **Lesson:** palette tokens have intended *contexts* — `surface` is for canvas obstacles, not for text on dimmed overlays. Use `palette.background` (the brightest token) for overlay-anchored text.
- **Sharing "42" as the menu row label.** PR #52 and PR #53 both reviewed and merged with the arcade menu row reading "42". Spoiled the punchline from the entry point. PR #54 was a single-day-later cleanup. **Lesson:** player-eyes review is a different pass from playtest review — text that pre-spoils a joke only surfaces after living with the artifact, not during build.
- **The captured-flavor bug in Pong's initial draft.** Assistant initially captured the random `flavor` string in a `useState(() => flavor)` lazy-init at component mount — when phase was still 'playing' and outcome was 'fail' at score 0-0. Would have shown a fail flavor even on a win. Caught before commit; switched to `useMemo(() => pickFlavor(...), [outcome])` so the pool draw fires once per outcome transition. **Lesson:** `useState(() => fn())` only fires once — wrong primitive when the value depends on state that changes after mount; `useMemo` over the changing dep is right.
