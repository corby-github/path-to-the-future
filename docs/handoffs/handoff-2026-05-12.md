## Key Decisions and Why

### Analytics provider: GoatCounter cloud
- **What:** Use GoatCounter cloud (free for non-commercial) as the §24 provider.
- **Why:** No cookies → no consent banner, ~3KB script, ad-blocker resistant. Zero backend on the user's side.
- **Alternative considered:** Plausible (paid cloud, more polished), Cloudflare Web Analytics (requires DNS proxy through CF), GA4 (explicitly rejected — cookie banner + heavy script). Self-hosting was explicitly clarified as NOT what GoatCounter cloud requires.

### Scoreboard trust model: graffiti board
- **What:** When §25 ships, accept that scores are client-computed and trivially cheatable. No replay verification.
- **Why:** Matches the project's spirit (now memorialized in `project_spirit_and_motivation.md`). User: "even the best controls blow this up and that's not what we are going for." Median submission is probably honest; cheaters are part of the texture.
- **Alternative considered:** Server-side replay (would need to port `computeScore` + effect engine to a backend), HMAC signing (security theater — key extractable from bundle).

### XP economy shape: layered (baseline + minigame + decision-encoded)
- **What:** +50 baseline / 250/100/25 minigame / +300 promotions / +150 stretches.
- **Why:** User pushed back on baseline-only ("50 feels slow"). Layered shape rewards big career moments while keeping steady accumulation feel. Math lands a "go for it" Novice at mid-Skilled by month 120; "play it safe" at low-Skilled.
- **Alternative considered:** Per-month base in `completeMonth` (deferred — time-skip months would lose XP); tag-based bonuses (rejected — explicit `xp` effect key is finer authoring control).

### Identifiability via `data-*`, NOT CSS classes
- **What:** Sprinkle `data-component` / `data-region` / `data-action` attributes across the UI. Inline styles unchanged.
- **Why:** Palette/era-mood model is JS-driven from React context. CSS classes would require either hardcoding colors (loses career-pack flexibility) or building a parallel CSS-variable system mirroring the React context for no real gain.
- **Alternative considered:** className with palette in CSS variables (rejected as parallel system), IDs (rejected — components render multiple times).

### Process rules saved as feedback memories
- **Update the design doc in the same PR as the code change.** Triggered by PR #35 shipping without §8/§9 updates → user flagged → memory saved → applied immediately to backfill via #36 and forward in #37.
- **Add `data-*` from the first commit, not retroactively.** Triggered by PR #29 being a 20-file sweep that "should have been there all along."
- **Skip verbose browser smokes.** User runs UI checks themselves; verify gate + reasoning is enough. The Claude_Preview MCP server's PostToolUse hook keeps reminding to `preview_start` — the memory overrides the hook reminder; the user opted to "live with it" rather than disable the MCP server.

## Dead Ends

- **Automated browser smoke for decision-commit XP path.** Tried to use `preview_eval` to simulate arrow-key movement to walk the player to a decision door, trigger the modal, commit an option, observe XP increment. Synthetic `KeyboardEvent('keydown', { key: 'ArrowRight' })` and `key: 'd'` were dispatched on `window` but didn't reach the movement engine — either the input ref was guarded by something I couldn't see, or the eval thread held up the rAF loop, or both. Hit a 30-second timeout and gave up. **Lesson:** the movement engine is hard to drive from outside; for future verification, either expose `store` on `window` in dev mode or accept that decision-flow verification is manual.

- **`preview_eval` verbose chains slowing the loop.** Walked through the init flow + minigame using multi-step preview_eval IIFEs. User explicitly called it out as too slow: *"I can run those checks, yours seem too verbose."* Saved as feedback memory; default going forward is to skip browser smoke entirely (verify gate + reasoning) and ask before doing real walkthroughs.

- **Adding interactable label below `[E]` hint at low contrast.** Initial PR #37 render put the label directly under the `[E] talk` text at `palette.inkMuted`. User intercepted before commit: too light, and the label should sit under the *object*, not under the `[E]` hint. Iterated: label moved below the sprite, bumped to `palette.ink` weight 600. **Lesson:** for any new player-facing UI element, default to higher-than-expected contrast and bracket the sprite with above/below labels rather than stacking them.
