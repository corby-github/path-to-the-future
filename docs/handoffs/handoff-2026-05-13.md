## Key Decisions and Why

### Generator is SSR via `tsx` + `react-dom/server`, not a hand-written SVG stringifier
**Decision:** Re-use the React icon components directly via SSR rather than authoring a parallel "rendered string" map for the previews.
**Why:** The icons already exist and consume `palette` as a prop — no `useCareerPack()` hook in their bodies. SSR with the SWE palette gives the previews the exact same SVG the modal ships. A parallel string map would drift the moment an icon changed.
**Alternative considered:** Manual SVG-string extraction or playwright-driven screenshotting. Both higher-friction and would drift from source.

### `docs/research/` as a new artifact type, separate from `docs/logs/` and `docs/handoffs/`
**Decision:** Create a new top-level docs subdirectory for durable project-level observations / principles / findings.
**Why:** Existing dirs are scoped: `logs/` is sprint records, `handoffs/` is session bridges, `text-previews/` and `icons-previews/` are review surfaces. Cross-cutting principles ("AI/human review asymmetry," "context bias after compact") don't fit any of those; they need their own home so they can be cross-referenced over time.
**Alternative considered:** Bury them in the canonical design doc. Rejected — design doc is about the *product*; these are about the *process*.

### `/research` as a *universal* skill, not project-local
**Decision:** Lives at `~/.claude/skills/research/`, writes to `docs/research/` relative to cwd.
**Why:** Matches the `/punch` and `/usage` pattern; works in any project; one skill body to maintain.
**Alternative considered:** Project-local at `.claude/skills/`. Rejected — fragmentation across projects.

### Boundary scheme is paired (open/close), not single-row marker
**Decision:** Time-log boundaries use `boundary-open` + `boundary-close` event rows that bracket a sub-range.
**Why:** A single row marks *when* but not *for how long*. User flagged this explicitly after I shipped the first single-marker version. Pairs let `session-process-log` compute exact `process_total` and subtract it.
**Alternative considered:** Single marker with a duration column. Rejected because the time-log row schema is "one event per row" — adding a duration to a single row breaks that.

### `session-process-log` compares traditional-team estimate against `product_duration`, not `sprint_duration`
**Decision:** When computing time-savings, the denominator is product wall time only.
**Why:** Crediting AI velocity with hours spent editing Claude-side skills is a category error and would inflate the headline number. The boundary scheme exists exactly to prevent this.
**Alternative considered:** Roll everything into one number and footnote the split. Rejected — footnotes get skipped.

### Voice-checkpoint protocol stays "5+5 starter batch" before scaling
**Decision:** When the homeschool icon sprint starts, author the first 5 decision-pool icons + first 5 event-pool icons, run the generator, voice-check before authoring the remaining ~57.
**Why:** Established on PR #56 (Phase 1) and PR #60 (Phase 2). Cheap insurance against voice/style drift across 67 icons.

## Dead Ends

### `npm run gen:previews` first attempt: "ReferenceError: React is not defined"
**What was tried:** Just running `tsx scripts/generate-previews.tsx` with no tsconfig.
**Why it failed:** tsx (the executor) defaults to esbuild's `jsx: "transform"` which compiles JSX to `React.createElement(...)` — but `React` isn't imported at the top level (the icons use the automatic JSX runtime via the project's `tsconfig.app.json`).
**Resolved by:** Creating `scripts/tsconfig.json` with `"jsx": "react-jsx"` + `"jsxImportSource": "react"` and passing `--tsconfig scripts/tsconfig.json` to tsx.

### First `/usage` snapshot after compact: estimated ~65% / ~540k, actual 14% / 143.3k
**What was tried:** Heuristic estimation from the usage-log skill's "long session 50–80%" band.
**Why it failed:** Post-`/compact`, the visible window is small even if the session *feels* long. My heuristic pattern-matched the felt narrative, not the actual loaded token count.
**Resolved by:** Patched `/usage` skill body with a "Post-compaction correction" subsection. Also captured as research artifact at `docs/research/context-estimation-bias-after-compact.md`.

### First boundary scheme: single-row `boundary` event (no end marker)
**What was tried:** A single `Event = boundary` row at 10:45 with a long note saying "everything after this is process tooling."
**Why it failed:** User asked the right question — "looking at it by itself it's unclear how MUCH time was used for process tooling." Single-edge marker doesn't yield a duration.
**Resolved by:** Replaced with paired `boundary-open` + `boundary-close` rows. Formalized in `/punch` skill body.
