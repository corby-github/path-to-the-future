## Key Decisions and Why

### Renumber monthIds 1..120 → 1..70 (vs. keep 120 with skip markers)
**Decision:** Rewrite `months.json` to 70 entries with sequential ids. `currentMonth` advances 1→2→3..→70.
**Why:** Cleanest mapping. Every consumer (replay door, completed-months ledger, save migration) keeps using `id+1` semantics without skip-aware lookup. The slot table `SLOT_TO_MONTH_NUM = [1, 2, 4, 6, 8, 10, 12]` localizes the calendar nonlinearity in one constant.
**Alternative considered:** Keep ids at 1..120 and add `roomType: 'skip'` or filter-on-advance. Rejected — would have intrusively touched `completeMonth`, replay logic, history/score formula, etc.

### Branch PR2 from `main` (not stack on PR1)
**Decision:** Both PRs target `main` independently.
**Why:** `feedback_no_stacked_prs.md` rule. PR2 is ~95% orthogonal to PR1 — they touch different files in the generator stack. Only collision is the design-doc change-log row order (a 30-second rebase whichever lands second).
**Alternative considered:** Stack PR2 on `feat/half-length-playthrough` to share the new `LAST_MONTH_ID = 70`. Rejected per the memory rule; the share isn't needed for PR2's framework (year-driven mix doesn't depend on the calendar refactor).

### Synthetic finale decision id rename: `finale-month-120` → `finale-month`
**Decision:** Drop the monthId from the synthetic decision id when bumping `FINALE_MONTH_ID` to 70.
**Why:** The id is an opaque string keying the icon registry and the hardcoded DecisionRoom path. Encoding `120` made it look like the value mattered; the rename decouples the id from any future monthId shift.
**Alternative considered:** Rename to `finale-month-70`. Rejected — same problem one rev later.

### Pong minigame: Mar 2026 → Apr 2026 (not Feb 2026)
**Decision:** When March 2026 stopped being a playable month, the closest replacement was Apr 2026 (id 45), one month later.
**Why:** Forward-shift preserves the "ai-shift era, year 2026" feel and lands on the closest **playable** calendar position. Backward-shift to Feb 2026 (id 44) was in-era too but felt like rewinding history.
**Alternative considered:** Different year entirely. Rejected — minigame scheduling was already tuned for narrative arc.

### Resolve PR2 distribution spec ambiguity by smoothing to a monotonic curve
**Decision:** User spec had three conflicting entries for 2028 (`50/50 medium/hard`, `80% hard 20% medium`, plus a "2030 = 50/50 hard/expert" row in a 2020–2029 game). Resolved to:
- 2027 = 50% medium / 50% hard
- 2028 = 20% medium / 80% hard
- 2029 = 50% hard / 50% expert
**Why:** Produces a monotonic difficulty curve from `simple` (2020) → `hard/expert` (2029). The "2030" entry was almost certainly meant to be 2029. Called out in the PR description so the user can override the mix with a single constant edit.
**Alternative considered:** Surface the ambiguity via AskUserQuestion. Held back — the resolution is reversible (one constant in `layouts.ts`) and the broader scope question had already burned an AskUserQuestion. Tested when user reviews PR #79.

### `STATE_VERSION` bump on PR1 (discard old saves)
**Decision:** `1.6.0 → 1.7.0`.
**Why:** Old saves with `currentMonth > 70` no longer map to a valid month entry; would mis-render the finale or crash on lookup. Bump matches the existing "in-dev save invalidation" pattern (1.4.0 / 1.5.0 / 1.6.0 all did the same).
**Alternative considered:** Migration shim that clamps stale `currentMonth` to 70. Rejected as overengineering for a project still pre-v1-ship.

### Don't bump `XP_PER_DECISION` in PR1
**Decision:** Leave XP economy alone in PR1 even though halving the playthrough cuts expected XP from `~110 × 50 = 5500` to `~60 × 50 = 3000` (target was ~6000 baseline).
**Why:** XP rebalance depends on the new run length AND the new tier mix (PR2). Tuning before PR2's framework is fully populated with content would over-fit. Called out as a follow-up in PR #78's body.
**Alternative considered:** Bump `XP_PER_DECISION` to 90 in PR1. Rejected — premature, prefer one tuning pass after rooms settle.
