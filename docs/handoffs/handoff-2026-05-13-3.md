## Key Decisions and Why

### Match doc to code, not code to doc (PR #70)
**Decision:** Reconcile §16 + §26 career roster *down* to the 5 entries in `careers.ts`, not up to the aspirational 9 in v2.0.
**Why:** The picker is what ships; the doc should describe what ships. v2.0 nine-career roster was design intent, not a build commitment. Wider tonal aspiration kept in surrounding prose.
**Alternative considered:** Add 4 missing locked entries back to `careers.ts` (rejected as overengineering); two-table approach (rejected as confusing).

### Pack-aware template filter via `packs?` field on the template (PR #71)
**Decision:** Optional `packs?: readonly string[]` on `LayoutTemplate`. Universal if omitted; pack-restricted if listed.
**Why:** Mirrors `manifest.statLabels` / `manifest.classLabels` patterns the user already knows. Smallest delta over the existing TS-array layout system. Full JSON-per-pack refactor is the post-v1 architecture move; this shape keeps the door open without committing.
**Alternative considered:** Per-pack JSON manifests (cleaner architecture, ~2-3× the work). Exclude-list on each variant. Just add universal templates without filtering (lets office cubicles show up in homeschool runs).

### Maze: pure zigzag, no dead-end stubs (PR #72)
**Decision:** Ship the maze with only the 4 zigzag walls. Skip dead-end horizontal stubs.
**Why:** `PLAYER_RADIUS = 14` (diameter 28). Dead-end stubs that constrain corridors to <60 px risk soft-locking the player. User said *"do the best you can, I'll verify the path works"* — conservative geometry is the right call for first-pass.
**Alternative considered:** 2 dead-end stubs for "more maze-like" feel. Held for follow-up if pure zigzag feels too easy after playthroughs.

### ProfileModal: player-name editable, kids disabled with tooltip (PR #75)
**Decision:** Ship the profile card with player-name inline-edit; kid edit buttons **disabled** with "Coming soon" tooltip. File issue #76 for the full kid-name interpolation sprint.
**Why:** Kid names are hardcoded 74 times across `public/careers/homeschool-parent/*.json`. Editing them in the modal without rewriting the content creates a mismatch (modal says "Lily," game still says "Hazel"). The full interpolation sprint is ~3-4h of mechanical work — its own focused PR.
**Alternative considered:** Full kid-name interpolation in this PR (rejected as scope). State-only kid edit without content rewrite (rejected as confusing UX).

### Closed-union typecheck enforces icon-registry exhaustiveness (PR #73)
**Decision:** Type `MINIGAME_ICONS` as `Record<MinigameVariant, ModalIconComponent>` (not `Partial<...>`).
**Why:** Future variants will fail the build until they're registered. Prevents the exact orphan-art bug this PR was closing — `npm run verify` becomes the enforcement.
**Alternative considered:** `Partial<Record<...>>` with `PlaceholderIcon` fallback (less guard-rail-y). Runtime assertion (more boilerplate).

### Stacker speeds as a literal tuple, not a formula (PR #74)
**Decision:** `BLOCK_SPEEDS = [480, 620, 520, 680, 560] as const`.
**Why:** User asked to *"alternate the speed"* — the literal tuple makes the alternation visible at the const-declaration site. A formula would hide it.
**Alternative considered:** Formula with odd-block multiplier. Random per session.

### Force-with-lease over plain force-push (PR #73 rebase)
**Decision:** Use `git push --force-with-lease` when republishing the rebased branch.
**Why:** Per CLAUDE.md's "destructive ops require confirmation" rule, asked the user first. `--force-with-lease` refuses if anyone else pushed in the meantime — safer than plain `--force` for a published branch.
**Alternative considered:** `git merge origin/main` into the branch (no rewrite, but adds a merge commit). Close PR + reopen (loses the existing PR thread).

**Lesson:** Stable taxonomy beats expressive taxonomy. Memory codifies the 3 buckets.
